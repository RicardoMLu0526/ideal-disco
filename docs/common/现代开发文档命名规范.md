# 现代开发文档命名规范

## 1. 概述

本文档定义了现代软件开发项目中文档的命名规范，旨在提高文档的一致性、可读性和可维护性。适用于各类技术文档，包括但不限于需求文档、技术方案、API文档、测试文档等。

## 2. 命名原则

### 2.1 清晰性
- 命名应准确反映文档内容和用途
- 使用具体、明确的词汇，避免模糊和歧义
- 优先考虑可读性，而非简洁性

### 2.2 一致性
- 在整个项目中保持命名风格一致
- 遵循团队或组织已有的命名约定
- 对于相似类型的文档，使用相同的命名模式

### 2.3 规范性
- 使用标准的文件扩展名
- 遵循行业通用的命名惯例
- 符合版本控制系统的要求

### 2.4 可搜索性
- 包含关键信息，便于搜索和识别
- 合理使用前缀和后缀，提高分类效果
- 避免使用特殊字符，确保跨平台兼容性

## 3. 基础命名规则

### 3.1 文件名结构
```
[前缀]-[主体内容]-[后缀].[扩展名]
```

### 3.2 命名约定
- **使用小写字母**：统一使用小写字母，避免大小写混淆
- **单词分隔**：使用连字符(`-`)分隔单词，避免使用下划线(`_`)或空格
- **避免特殊字符**：不使用除连字符外的特殊字符
- **长度限制**：文件名长度建议不超过50个字符

### 3.3 文件扩展名
| 文档类型 | 推荐扩展名 | 说明 |
|---------|-----------|------|
| 通用文档 | .md | Markdown格式，推荐使用 |
| 技术文档 | .md | 技术方案、设计文档等 |
| API文档 | .md | API接口说明文档 |
| 需求文档 | .md | 产品需求、功能需求等 |
| 测试文档 | .md | 测试计划、测试用例等 |
| 配置文件 | .yaml, .yml, .json, .toml | 配置相关文档 |
| 脚本文件 | .sh, .py, .js | 自动化脚本 |

## 4. 具体文档类型命名规范

### 4.1 需求文档

#### 4.1.1 产品需求文档(PRD)
```
prd-[功能模块]-[版本号].md
```

**示例**：
- `prd-user-authentication-v1.0.md`
- `prd-payment-system-v2.1.md`

#### 4.1.2 技术需求文档
```
tech-requirement-[功能模块]-[版本号].md
```

**示例**：
- `tech-requirement-data-sync-v1.0.md`
- `tech-requirement-cache-system-v1.2.md`

### 4.2 技术文档

#### 4.2.1 技术方案文档
```
solution-[技术领域]-[功能模块].md
```

**示例**：
- `solution-architecture-microservices.md`
- `solution-database-sharding.md`

#### 4.2.2 架构设计文档
```
arch-[架构层次]-[模块名称].md
```

**示例**：
- `arch-backend-order-service.md`
- `arch-frontend-user-interface.md`

#### 4.2.3 技术选型文档
```
tech-selection-[技术类别].md
```

**示例**：
- `tech-selection-database.md`
- `tech-selection-frontend-framework.md`

### 4.3 API文档

#### 4.3.1 REST API文档
```
api-rest-[资源名称]-[版本号].md
```

**示例**：
- `api-rest-users-v1.md`
- `api-rest-products-v2.md`

#### 4.3.2 GraphQL API文档
```
api-graphql-[功能模块].md
```

**示例**：
- `api-graphql-user-management.md`
- `api-graphql-order-processing.md`

### 4.4 测试文档

#### 4.4.1 测试计划
```
test-plan-[测试类型]-[版本号].md
```

**示例**：
- `test-plan-integration-v1.0.md`
- `test-plan-performance-v2.0.md`

#### 4.4.2 测试用例
```
test-case-[功能模块]-[编号].md
```

**示例**：
- `test-case-login-flow-001.md`
- `test-case-payment-process-002.md`

### 4.5 部署与运维文档

#### 4.5.1 部署文档
```
deploy-[环境]-[服务名称].md
```

**示例**：
- `deploy-production-api-gateway.md`
- `deploy-staging-frontend-app.md`

#### 4.5.2 运维文档
```
ops-[主题]-[服务名称].md
```

**示例**：
- `ops-monitoring-backend-services.md`
- `ops-backup-database.md`

### 4.6 开发规范文档

```
规范-[规范类型].md
```

**示例**：
- `规范-code-style.md`
- `规范-git-workflow.md`
- `规范-命名约定.md`

## 5. 版本控制相关命名

### 5.1 版本号格式
```
v[主版本号].[次版本号].[修订号]
```

**示例**：
- `v1.0.0`
- `v2.1.3`

### 5.2 文档版本管理
- 对于重要文档，建议在文件名中包含版本号
- 对于频繁更新的文档，可使用日期作为版本标识
- 建议在文档内部维护版本历史记录

**日期格式**：`YYYY-MM-DD`

**示例**：
- `api-spec-v2.0-2024-01-15.md`
- `deployment-guide-2024-02-01.md`

## 6. 国际化文档命名

### 6.1 语言标识
- 使用ISO 639-1标准的语言代码
- 语言代码应作为文件名的一部分

### 6.2 命名格式
```
[文档名称]-[语言代码].[扩展名]
```

**示例**：
- `user-guide-en.md`
- `api-reference-zh.md`
- `technical-specification-ja.md`

## 7. 目录结构建议

为了更好地组织文档，建议采用以下目录结构：

```
docs/
├── requirements/           # 需求文档
├── design/                 # 设计文档
├── api/                    # API文档
├── test/                   # 测试文档
├── deploy/                 # 部署文档
├── ops/                    # 运维文档
├── standards/              # 规范文档
└── README.md               # 文档说明
```

## 8. 最佳实践

### 8.1 命名示例

| 文档类型 | 推荐命名 | 说明 |
|---------|---------|------|
| 产品需求 | `prd-user-registration-v1.0.md` | 清晰标识文档类型、功能和版本 |
| 技术方案 | `solution-database-design.md` | 明确文档内容和用途 |
| API文档 | `api-rest-products-v2.md` | 包含API类型、资源和版本 |
| 测试用例 | `test-case-checkout-flow-001.md` | 包含测试类型、功能和编号 |
| 部署文档 | `deploy-production-backend.md` | 明确环境和服务 |

### 8.2 常见错误示例

| 错误命名 | 问题 | 正确命名 |
|---------|------|---------|
| `文档1.md` | 过于模糊，无具体信息 | `prd-login-function-v1.0.md` |
| `API Doc.md` | 使用空格和大小写混合 | `api-reference-users.md` |
| `tech_spec_database.md` | 使用下划线分隔 | `tech-spec-database.md` |
| `测试计划v2.0.md` | 中英文混合，格式不一致 | `test-plan-v2.0.md` |
| `deployment-guide-20240115.md` | 日期格式不标准 | `deployment-guide-2024-01-15.md` |

## 9. 工具支持

### 9.1 推荐工具
- **Markdown编辑器**：VS Code + Markdown插件、Typora、Obsidian
- **文档管理**：GitBook、Docusaurus、MkDocs
- **版本控制**：Git、SVN

### 9.2 自动化工具
- 考虑使用脚本自动化文档命名和组织
- 利用CI/CD流程自动生成和更新文档
- 使用文档生成工具从代码注释生成API文档

## 10. 总结

良好的文档命名规范是项目成功的重要组成部分。通过遵循本文档定义的命名原则和具体规则，可以：

- 提高文档的可发现性和可管理性
- 减少团队成员之间的沟通成本
- 确保文档的一致性和专业性
- 便于版本控制和历史追溯

建议团队成员共同遵守这些规范，并根据项目的具体情况进行适当调整和补充。
